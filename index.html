Empower your Odoo development workflow with complete, standard & cloud native tooling which is easy to integrate with CI/CD and ready to deploy.

Bootstrap your project

$ WORKDIR=$(pwd)/dev/
$ PROJECT=my-project
$ DEFAULT_BRANCH=11.0

$ mkdir -p "${WORKDIR}" && \
  cd "${WORKDIR}" && \
  git clone https://github.com/xoe-labs/dockery-odoo-scaffold.git "${PROJECT}"  && \
  cd  "${PROJECT}" && \
  git submodule add -b "${DEFAULT_BRANCH}" https://github.com/odoo/odoo.git vendor/odoo/cc

# If you have access to enterprise code
$ git submodule add -b "${DEFAULT_BRANCH}" git@github.com:odoo/enterprise.git vendor/odoo/ee

Your project environment

$ sed -i "s/{{PROJECT}}/${PROJECT}/" ./Dockerfile ./.env && \
  sed -i "s/{{DEFAULT_BRANCH}}/${DEFAULT_BRANCH}/" ./Dockerfile ./.env

## Review and customize your Dockerfile
$ nano ./Dockerfile

## Review and customize your .env file
$ nano ./.env

Launch using docker-compose

# Make use of the provided scripts
$ chmod -R +x ./scripts

# Build the images
$ ./scripts/run-images.sh --pull

# Initialize db server (first run)
$ docker-compose up odoo


# IMPORTANT: Place at least one module in the src folder.
# Or: Jump over to scaffolding before launching.

# Launch your dev instance, Odoo is served on port 80
$ docker-compose up odoo

# Get the admin password
$ cat .adminpwd

Scaffold your first module

# Apply patches to your workdir
$ ./scripts/run-patches.sh

# Scaffold
$ docker-compose run -u $UID scaffold "My next big module"

Poke arround, test and explore

# Run a Odoo shell
$ docker-compose run shell -d <DATABASE>

# Test a module
$ docker-compose run tests -d <DATABASE> -i <MODULE>

# Version 12.0 and above:
# View screenshots taken from the js tests
# Just own the output folder by the "magic" id
$ sudo chown -R 9001 _testlogs

Usage Scenarios

There are two main usage scenarios.


Client Project


Characteristics:

  • Module dependencies are vendored in vendor/<ORG>/<REPO>/<MODULE>1.
  • src is used for custom development.
  • It is best practice to version control configuration and meta-data.
  • Such custom_config module satisfies the min. module condition.

1 Use sparse-checkout to only get the modules you need.

Odoo Dev Team


Characteristics:

  • Stable repos are vendored in vendor/<MYORG>/<REPO>.
  • Module dependencies, if any, same as for Client Projects.
  • src is used for pre-release module development.
  • src is a shared workdir on the team’s remote repo.1
  • The team uses feature branches to develop new modules.
  • After beta acceptance, the module is moved to it’s vendored repo 2.
  • Further maintenance is done on that target repo.

1 Nice: src modules override their namesakes in vendored repos, as they are placed into .../090, the last (-> first) addon path.

2 Use “Initial commit” to squash development history or git subtree commands to preserve it or, to get fancy, adopt a git subrepo workflow to keep disconnected histories in sync during beta phase.

Remember:

  • Best use a git submodule workflow for your vendored repositories.
  • For sparse-checkout on submodules, start with this stackoverflow answer.
